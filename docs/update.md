## 一、练习-复习-考试流程改进

### 问题 1：强制加入复习计划，无法退出

**现状**：只要 `recordProgress(id, true)` 被调用（首次正确），文章就自动进入艾宾浩斯复习计划，没有"不加入"或"移除"的选项。用户可能只是随便听一听，不想背负后续7轮复习压力。

| 维度 | 分析 |
|------|------|
| **问题本质** | 复习计划是"隐性契约"，用户没有知情同意权 |
| **用户影响** | 复习列表越来越长，心理负担重，最终放弃整个复习体系 |
| **改进方案** | ① 练习完成后底部出现"📌 加入复习计划"按钮（替代自动加入）<br>② 复习列表中每篇增加"移除"操作<br>③ 设置页增加"自动加入复习"开关（默认关） |

---

### 问题 2：单篇"查看答案"≠"完成复习"，练习数据丢失

**现状**：用户听完音频→查看答案→直接关闭弹窗，这次练习不会被 `recordProgress` 记录。只有点"✅ 完成复习"才记录。

| 维度 | 分析 |
|------|------|
| **问题本质** | "看答案"和"记录进度"是两个动作，但用户心智模型中它们是一体的 |
| **用户影响** | 练习次数统计偏低，用户觉得"我明明练了但数字没变" |
| **改进方案** | ① 查看答案时自动调用 `recordProgress`，"完成复习"仅负责关闭弹窗<br>② 或：关闭弹窗时检测是否已查看答案，若已查看则弹确认"本次练习尚未记录，确认关闭？"<br>③ `listeningLog.count` 在音频播放完毕时即+1（不论是否查看答案），`log.c` 仍等用户确认 |

---

### 问题 3：单篇"完成复习"固定 correct=true，间隔重复永不倒退

**现状**：`completeListening()` 硬编码 `recordProgress(articleId, true)`。无论用户实际答得多差，复习进度都在往前推。艾宾浩斯体系的核心——"答错则回退"——完全失效。

| 维度 | 分析 |
|------|------|
| **问题本质** | 单篇模式没有"判定对错"的闭环，正确率计算出来后无处使用 |
| **用户影响** | 复习计划名存实亡——每篇只需7次点击"完成复习"就"已掌握"，哪怕每次都答错 |
| **改进方案** | ① 单篇查看答案后，用已有的 `wordPct`（听写）或 `pct`（填空）来判定 correct：`wordPct ≥ 80%` → true，否则 false<br>② 答错时 `log.c` 不变，`log.next` 回退到更短间隔（如当前在第4轮答错，回退到第2轮的间隔）<br>③ 连续答错2次则 `log.c -= 1`（最低为0），实现真正的"间隔递减" |

**回退策略对照表**：

| 当前轮次 | 答错回退到 | 间隔示例 |
|---------|-----------|---------|
| c=1（第1轮） | 不回退，next重设为明天 | 1天 |
| c=2（第2轮） | 回退到 c=1 | 1天 |
| c=3（第3轮） | 回退到 c=1 | 1天 |
| c=4（第4轮） | 回退到 c=2 | 2天 |
| c=5（第5轮） | 回退到 c=2 | 2天 |
| c=6（第6轮） | 回退到 c=3 | 4天 |

### 问题 4：批量考试选题无智能，待复习文章被忽略

**现状**：`startQuickListen(count)` 从全量池随机抽取，不考虑文章是否需要复习、是否薄弱。

| 维度 | 分析 |
|------|------|
| **问题本质** | 考试和复习是两个割裂的系统，没有联动 |
| **用户影响** | 待复习文章越积越多，但用户每天只做随机考试，复习列表从不清理 |
| **改进方案** | ① 默认优先从待复习列表抽题，不足再从未练习的补<br>② 首页"听N篇"按钮改为"🎧 复习N篇"（当有待复习时），无待复习时才显示"随机听N篇"<br>③ 增加"薄弱优先"策略：正确率低的优先出现 |

**选题优先级**：

```
待复习 > 最近答错过 > 未练习 > 已掌握
```

---

### 问题 5：跳过 = 答错，没有"中间态"

**现状**：跳过题目 `correct=false`，与真正答错完全等同，扣分也一样。

| 维度 | 分析 |
|------|------|
| **问题本质** | "我不会"和"我不想做"是两种完全不同的意图，系统没有区分 |
| **用户影响** | ① 遇到超纲文章只能跳过，却被惩罚<br>② 统计数据失真——跳过和答错混在一起 |
| **改进方案** | ① 跳过 = 不计分不扣分，`correct=null`，该题不进入 `recordProgress`<br>② 批量结果中跳过用 ⏭️ 单独显示，不标红也不标绿<br>③ 如果连续跳过3题，弹提示"是否切换到更简单的分类？" |

## 二、奖励与扣分机制改进

### 问题 6：单篇三种模式奖励无差异

**现状**：单篇模式无论听写/填空/拼读，完成都固定 +5⭐。但三种模式的难度差距极大——听写需要盲写全文，拼读只需跟读。

| 维度 | 分析 |
|------|------|
| **问题本质** | 奖励没有反映付出，低效激励 |
| **用户影响** | 理性用户会选最简单的拼读模式刷分 |
| **改进方案** | 单篇奖励与批量对齐：听写+20、填空+10、拼读+5 |

**单篇/批量奖励统一表**：

| 模式 | 正确 | 错误 | 跳过 |
|------|------|------|------|
| 🎧 听写 | +20⭐ | -5⭐ | 0⭐ |
| ✏️ 填空 | +10⭐ | -3⭐ | 0⭐ |
| 🗣️ 拼读 | +5⭐ | 0⭐ | 0⭐ |

---

### 问题 7：星星可能为负，无保底

**现状**：批量考试中 `u.stars += starReward`，如果全答错，10篇听写可以扣到 -50⭐。负分极度挫败。

| 维度 | 分析 |
|------|------|
| **问题本质** | 惩罚无上限，违背"鼓励学习"的产品初衷 |
| **改进方案** | ① 星星最低为0，`u.stars = Math.max(0, u.stars + starReward)`<br>② 或：扣分封顶——单次考试最多扣 10⭐，超出不扣 |

---

### 问题 8：没有连续/每日激励

**现状**：只有单次行为的奖扣，没有"连续练习N天""今日目标"等长期激励。

| 维度 | 分析 |
|------|------|
| **问题本质** | 缺少留存机制，用户容易断更 |
| **改进方案** | ① 连续打卡奖励：连续3天+20⭐，7天+50⭐，30天+200⭐<br>② 每日目标：今日练5篇额外+10⭐<br>③ 待复习清零奖励：今天全部复习完+30⭐ |

---

### 问题 9：查看答案评分与进度记录完全脱节

**现状**：单篇听写模式中，`revealAnswer()` 计算了 `wordPct`（单词匹配率）并展示给用户，但这个评分仅作为 UI 展示，**没有任何下游消费**——不影响 `recordProgress` 的 correct 参数（固定 true），不影响星星计算（固定 +5⭐），不影响间隔重复的回退逻辑。用户看到自己只得了 30%，点"完成复习"却和得 100% 完全一样。

| 维度 | 分析 |
|------|------|
| **问题本质** | 系统有了"打分"的能力，但分数在到达决策点之前就丢弃了 |
| **用户影响** | ① 用户看到差分却无任何后果，产生"分数没用"的感知<br>② 无法区分"蒙混过关"和"真正掌握"<br>③ 与批量模式形成体验割裂——批量有正确/错误的判定闭环，单篇没有 |
| **改进方案** | ① `completeListening()` 中读取当前 `wordPct`，`wordPct ≥ 80%` 时 `correct=true`，否则 `correct=false`，将评分传递给 `recordProgress`<br>② 填空模式同理，读取 `fillPct ≥ 60%` 为正确<br>③ 低分时弹窗提示"本次正确率较低（35%），建议明天再复习一次"，延迟关闭 2 秒让用户看到<br>④ 单篇星星也按得分梯度发放：wordPct ≥ 80% → +20⭐，60%~79% → +10⭐，<60% → 0⭐（不扣分，单篇模式本身是自由练习） |

**单篇评分→记录→奖励的完整流程（改进后）**：

```
查看答案 → revealAnswer() 计算 wordPct
  → 底部显示分数 + 等级提示
  → 点击"完成复习" → completeListening()
    → 读取 wordPct
    → wordPct ≥ 80% → correct=true, +20⭐
    → 60% ≤ wordPct < 80% → correct=true, +10⭐（降档奖励）
    → wordPct < 60% → correct=false, 0⭐
    → recordProgress(articleId, correct)
    → 如果 correct=false → 显示"💡 再练一次"按钮（不关闭弹窗）
```
